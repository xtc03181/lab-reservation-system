# 高校实验室预约申请与设备管理系统代码模块说明

本文档用于课程答辩复习，重点说明项目中每个模块的作用、实现方式，以及老师可能追问的问题。

## 一、项目整体架构

本项目采用前后端分离架构：

- 后端使用 Spring Boot 3，负责接口、业务逻辑、权限控制、数据校验和数据库访问。
- 前端使用 Vue 3 + Element Plus，负责页面展示、表单交互、菜单权限、接口调用和消息提示。
- 数据库使用 MySQL，保存用户、实验室、设备、预约、借用、公告、验证码和操作日志等数据。
- 数据访问层使用 MyBatis Plus，减少基础 CRUD 代码。

整体请求流程：

```text
用户在前端页面操作
-> Vue 页面调用 api/modules.js
-> Axios 请求拦截器自动携带 token、用户ID、角色
-> Spring Boot Controller 接收请求
-> RoleInterceptor 校验登录和角色权限
-> Service 执行业务规则
-> Mapper 操作 MySQL
-> ApiResponse 统一返回结果
-> 前端展示成功或错误提示
```

答辩时可以这样说：

> 我的系统采用前后端分离设计，前端只负责页面和交互，后端负责业务规则和数据库操作。所有接口返回统一的 ApiResponse 格式，前端通过 Axios 拦截器统一处理登录凭证和错误提示。

## 二、后端目录结构与作用

后端代码位于 `backend/src/main/java/com/gltu/labreservation`。

### 1. 启动类

文件：`LabReservationApplication.java`

作用：

- Spring Boot 项目入口。
- 通过 `@SpringBootApplication` 启动后端服务。
- 通过 `@MapperScan("com.gltu.labreservation.mapper")` 扫描 Mapper 接口，使 MyBatis Plus 可以访问数据库。

答辩可说：

> 这是后端启动入口，运行 main 方法后，Spring Boot 会加载 Controller、Service、Mapper、配置类等组件，并启动内嵌 Tomcat。

### 2. common 公共模块

目录：`common`

#### ApiResponse

作用：

- 统一接口返回格式。
- 成功返回 `code=200`，失败返回 `code=500` 和错误信息。

实现方式：

- 提供 `success(data)`、`success()`、`fail(message)` 三个静态方法。
- Controller 不直接返回原始对象，而是包装成 `ApiResponse`。

答辩可说：

> 统一返回格式后，前端可以只判断 code 是否为 200，不需要每个接口单独写不同的错误处理逻辑。

#### BizException

作用：

- 表示业务异常，比如“用户名或密码错误”“实验室时间冲突”“库存不足”。

实现方式：

- Service 层发现业务规则不满足时抛出 `BizException`。
- 由全局异常处理器统一转换成前端能识别的错误响应。

#### GlobalExceptionHandler

作用：

- 统一处理后端异常，避免异常直接暴露给前端。

实现方式：

- 使用 `@RestControllerAdvice` 和 `@ExceptionHandler`。
- 分别处理业务异常、参数校验异常、JSON 格式异常和其他系统异常。

答辩可说：

> 我没有让异常直接抛到页面，而是通过全局异常处理器统一包装，用户看到的是清晰的中文提示。

#### PasswordUtil

作用：

- 负责密码加密、密码匹配、旧密码升级、密码强度校验。

实现方式：

- 使用 SHA-256 对密码进行摘要，保存时加上 `SHA256:` 前缀。
- 登录时使用 `matches` 方法比较明文输入和数据库密码。
- 如果数据库里是旧的明文密码，会在登录成功后自动升级成 SHA-256 密文。
- 新密码要求至少 8 位，包含大写字母、小写字母、数字和特殊字符。

答辩可说：

> 密码没有直接明文保存，而是经过 SHA-256 摘要后保存。同时系统兼容旧数据，旧明文密码登录成功后会自动升级为加密格式。

#### TokenStore

作用：

- 登录成功后生成 token。
- 保存 token 与用户 ID、角色的对应关系。
- 后续请求通过 token 判断用户是否登录。

实现方式：

- 使用 `UUID.randomUUID()` 生成随机 token。
- 使用 `ConcurrentHashMap` 在内存中保存 token。

需要注意：

- 这是课程项目中的轻量实现。
- 如果是正式生产系统，可以换成 JWT 或 Redis 存储 token。

答辩可说：

> 这里我实现了一个轻量 token 登录机制。登录成功后返回 token，前端每次请求带上 token，后端拦截器根据 token 找到当前用户身份。

### 3. config 配置模块

目录：`config`

#### CorsConfig

作用：

- 解决前端 `localhost:5173` 调用后端 `localhost:8080` 时的跨域问题。

实现方式：

- 配置允许跨域请求，使前端可以正常调用后端接口。

#### WebMvcConfig

作用：

- 注册权限拦截器。

实现方式：

- 将 `RoleInterceptor` 添加到 Spring MVC 拦截器链中。
- 后端接口请求进入 Controller 前，先经过权限判断。

#### RoleInterceptor

作用：

- 校验用户是否登录。
- 校验请求头中的用户 ID、角色是否与 token 中保存的一致。
- 根据角色控制接口访问权限。

实现方式：

- 放行 `/auth/**` 登录和重置密码接口。
- 其他接口必须携带 `Authorization: Bearer token`。
- 管理员 `ADMIN` 拥有所有权限。
- 学生 `STUDENT` 可以查看信息、提交预约/借用、修改和取消自己的待审核申请。
- 教师 `TEACHER` 可以审核预约和借用申请。
- 操作日志和用户管理只允许管理员访问。

答辩可说：

> 系统有前端路由权限，也有后端接口权限。即使用户绕过前端直接请求接口，后端拦截器也会根据 token 和角色再次校验。

#### MyBatisPlusConfig

作用：

- 配置 MyBatis Plus 插件。
- 支持分页等扩展能力。

#### DatabaseInitConfig

作用：

- 数据库初始化相关配置。
- 课程项目中主要用于辅助本地开发。

### 4. entity 实体模块

目录：`entity`

实体类和数据库表基本一一对应：

- `User` 对应 `sys_user` 用户表。
- `Lab` 对应 `lab` 实验室表。
- `Equipment` 对应 `equipment` 设备表。
- `LabReservation` 对应 `lab_reservation` 实验室预约表。
- `EquipmentBorrow` 对应 `equipment_borrow` 设备借用表。
- `Notice` 对应 `notice` 公告表。
- `OperationLog` 对应 `operation_log` 操作日志表。
- `PasswordResetCode` 对应 `password_reset_code` 密码重置验证码表。
- `BaseEntity` 抽取公共字段，如 id、创建时间、更新时间、逻辑删除字段等。

答辩可说：

> Entity 层主要描述数据库表结构，MyBatis Plus 会根据实体类完成对象和数据库记录之间的映射。

### 5. mapper 数据访问模块

目录：`mapper`

作用：

- 负责数据库访问。
- 每个 Mapper 对应一个实体类。

实现方式：

- 继承 MyBatis Plus 的 `BaseMapper<T>`。
- 基础的新增、删除、修改、查询不需要手写 SQL。
- 复杂查询可以通过 `LambdaQueryWrapper` 构造条件。

答辩可说：

> Mapper 层只负责和数据库打交道，业务判断不写在 Mapper 里，而是放在 Service 层，这样分层更清晰。

### 6. dto 数据传输模块

目录：`dto`

作用：

- 接收前端请求或返回前端响应。
- 避免直接把实体类暴露给所有接口。

主要 DTO：

- `LoginRequest`：登录请求，包含账号和密码。
- `LoginResponse`：登录响应，包含 token 和用户信息。
- `ReviewRequest`：审核请求，包含审核状态、审核人、审核意见。
- `SendResetCodeRequest`：发送验证码请求，包含账号和邮箱。
- `ResetPasswordRequest`：重置密码请求，包含账号、邮箱、验证码、新密码。

答辩可说：

> DTO 用于接口参数传输，能让接口结构更明确，也能避免实体类字段全部暴露给前端。

### 7. controller 接口模块

目录：`controller`

Controller 负责接收 HTTP 请求，不直接写复杂业务逻辑，而是调用 Service。

#### AuthController 登录与密码重置

接口：

- `POST /auth/login`：用户登录。
- `POST /auth/send-reset-code`：发送邮箱验证码。
- `POST /auth/reset-password`：通过验证码重置密码。

实现流程：

```text
前端提交账号密码
-> AuthController.login
-> AuthService.login
-> 查询用户、校验密码
-> 生成 token
-> 返回 token 和用户信息
```

邮箱重置密码流程：

```text
输入账号和邮箱
-> 校验账号邮箱是否匹配
-> 生成 6 位验证码
-> 保存验证码、过期时间、使用状态
-> 通过 SMTP 邮件发送
-> 用户输入验证码和新密码
-> 校验验证码是否存在、未使用、未过期
-> 校验新密码强度和是否与旧密码相同
-> 更新密码并标记验证码已使用
```

#### UserController 用户管理与个人中心

接口：

- `GET /users`：查询用户列表。
- `GET /users/me`：查询当前登录用户信息。
- `POST /users`：新增用户。
- `PUT /users/{id}`：修改用户。
- `PUT /users/me`：修改个人资料。
- `PUT /users/me/password`：修改自己的密码。
- `DELETE /users/{id}`：删除用户。

实现要点：

- 返回用户信息时会把 `password` 设置为 `null`，避免密码传给前端。
- 新增用户如果不填密码，默认密码为 `Abc12345!`。
- 修改密码时需要校验原密码。
- 新密码必须满足强度规则，且不能和旧密码相同。
- 用户管理操作会记录操作日志。

#### LabController 实验室管理

作用：

- 管理实验室基础信息。
- 管理员可以新增、编辑、删除实验室。
- 学生可以查看开放的实验室信息。

实现方式：

- 主要调用 MyBatis Plus 的 `list`、`save`、`updateById`、`removeById`。
- 具体写权限由 `RoleInterceptor` 控制。

#### EquipmentController 设备管理

作用：

- 管理设备基础信息，包括设备名称、编号、所属实验室、型号、总数量、可借数量、状态。

实现方式：

- 管理员维护设备信息。
- 学生查看设备信息并发起借用申请。

#### LabReservationController 实验室预约

接口：

- `GET /lab-reservations`：查询预约列表。
- `POST /lab-reservations`：学生新增预约。
- `PUT /lab-reservations/{id}`：学生修改自己的待审核预约。
- `PUT /lab-reservations/{id}/cancel`：学生取消自己的待审核预约。
- `PUT /lab-reservations/{id}/review`：教师审核预约。

实现要点：

- 学生只能看到自己的预约记录。
- 教师和管理员可以看到全部预约记录。
- 新建预约时后端自动使用请求头中的当前用户 ID，防止前端伪造申请人。
- 只能修改或取消自己的待审核预约。
- 审核状态只允许 `APPROVED` 或 `REJECTED`。

预约冲突判断：

```text
同一个实验室
状态是 PENDING 或 APPROVED
已有预约开始时间 < 新预约结束时间
已有预约结束时间 > 新预约开始时间
```

如果满足以上条件，说明两个时间段有重叠，系统会拒绝提交。

答辩可说：

> 预约冲突判断不是简单比较开始时间是否相同，而是判断两个时间区间是否重叠。只要已有预约的开始时间早于新预约结束时间，并且已有预约的结束时间晚于新预约开始时间，就说明冲突。

#### EquipmentBorrowController 设备借用

接口：

- `GET /equipment-borrows`：查询借用列表。
- `POST /equipment-borrows`：学生提交设备借用申请。
- `PUT /equipment-borrows/{id}/review`：教师处理借用申请。

实现要点：

- 学生只能看到自己的借用记录。
- 提交借用时校验设备是否存在、借用数量是否大于 0、是否超过可借库存。
- 审核通过时扣减设备可借数量。
- 归还时恢复设备可借数量。
- 该方法使用 `@Transactional`，保证借用记录状态和设备库存同时成功或同时回滚。

答辩可说：

> 设备借用和库存变化是一个整体操作，所以我使用了事务。比如审核通过时既要修改借用状态，又要减少设备库存，如果中间出错就回滚，避免数据不一致。

#### NoticeController 公告管理

接口：

- `GET /notices`：查询公告。
- `POST /notices`：发布公告。
- `PUT /notices/{id}`：编辑公告。
- `PUT /notices/{id}/pin`：置顶公告。
- `PUT /notices/{id}/normal`：上架或取消置顶。
- `PUT /notices/{id}/offline`：下架公告。
- `DELETE /notices/{id}`：删除公告。

实现要点：

- 管理员可以查看所有公告，包括已下架公告。
- 非管理员只能查看已发布和置顶公告。
- 公告按状态和发布时间倒序排列，置顶公告更靠前。
- 公告内容前端使用 `white-space: pre-line` 保留换行。

#### OperationLogController 操作日志

作用：

- 管理员查看系统关键操作记录。

实现方式：

- 用户管理、个人中心、预约审核、借用审核、公告管理等操作完成后调用 `operationLogService.record`。
- 日志内容包括操作人、模块名、操作类型、详情和时间。

答辩可说：

> 操作日志用于追踪系统中的关键操作，比如谁发布了公告、谁审核了预约、谁修改了用户信息，方便管理员审计。

### 8. service 业务模块

目录：`service` 和 `service/impl`

Service 是系统业务规则的核心。

#### AuthServiceImpl

负责：

- 登录校验。
- token 生成。
- 邮箱验证码发送。
- 验证码重置密码。

关键实现：

- 使用 `UserMapper` 根据账号查询用户。
- 使用 `PasswordUtil.matches` 判断密码是否正确。
- 使用 `TokenStore.issue` 生成登录 token。
- 使用 `PasswordResetCodeMapper` 保存和查询验证码。
- 使用 `JavaMailSender` 发送邮件。

#### UserServiceImpl

负责：

- 新增用户。
- 修改用户。
- 修改密码。
- 校验角色和密码强度。

关键实现：

- 密码统一加密后保存。
- 新密码必须满足复杂度要求。
- 修改密码必须先校验原密码。

#### LabReservationServiceImpl

负责：

- 预约创建、修改、取消、审核。
- 校验实验室是否存在。
- 校验申请人是否存在。
- 校验时间是否合法。
- 校验预约是否冲突。

#### EquipmentBorrowServiceImpl

负责：

- 借用申请创建。
- 借用审核、驳回、归还。
- 校验库存。
- 审核通过扣减库存，归还恢复库存。

#### OperationLogServiceImpl

负责：

- 封装操作日志记录逻辑。

#### NoticeServiceImpl、LabServiceImpl、EquipmentServiceImpl

负责：

- 基础 CRUD 能力。
- 继承 MyBatis Plus 的 `ServiceImpl`，减少重复代码。

## 三、前端目录结构与作用

前端代码位于 `frontend/src`。

### 1. main.js

作用：

- Vue 应用入口。
- 注册 Pinia、Vue Router、Element Plus。
- 挂载根组件 `App.vue`。

### 2. App.vue

作用：

- 应用根组件。
- 通过 `<router-view />` 显示当前路由页面。

### 3. api/request.js

作用：

- 封装 Axios。
- 设置后端接口基础地址 `http://localhost:8080/api`。
- 请求时自动携带 token、用户 ID 和用户角色。
- 响应时统一处理 `ApiResponse`。

实现方式：

- 请求拦截器从 `localStorage` 读取 token 和 user。
- 响应拦截器判断 `result.code`。
- 如果后端返回“请先登录”，自动清除本地登录信息并跳转登录页。

答辩可说：

> 前端没有在每个页面重复写 token 逻辑，而是在 Axios 请求拦截器中统一添加请求头。

### 4. api/modules.js

作用：

- 集中定义所有后端接口方法。
- 页面不直接写 URL，而是调用封装好的函数。

示例：

- `login` 调用 `/auth/login`。
- `listLabs` 调用 `/labs`。
- `createReservation` 调用 `/lab-reservations`。
- `reviewBorrow` 调用 `/equipment-borrows/{id}/review`。

答辩可说：

> 我把接口统一封装在 modules.js 中，这样页面代码更简洁，后期如果接口地址变化也方便维护。

### 5. stores/user.js

作用：

- 使用 Pinia 保存登录状态。
- 保存 token 和当前用户信息。

实现方式：

- 登录成功后调用 `setLoginInfo`。
- token 和 user 同时写入 `localStorage`，刷新页面后仍能保持登录状态。
- 退出登录时清空 store 和 localStorage。

### 6. router/index.js

作用：

- 配置前端页面路由。
- 根据角色控制可访问页面。

实现方式：

- 每个页面路由设置 `meta.roles`。
- `beforeEach` 路由守卫判断是否登录。
- 未登录访问业务页面会跳转 `/login`。
- 已登录访问 `/login` 会跳转首页。
- 角色不匹配时跳回 `/dashboard`。

注意：

- 前端路由权限主要提升用户体验。
- 真正的接口安全仍由后端 `RoleInterceptor` 保证。

### 7. views 页面模块

#### Login.vue

作用：

- 登录页面。
- 忘记密码弹窗。
- 邮箱验证码重置密码。

实现方式：

- 登录成功后保存 token 和用户信息。
- 支持记住账号。
- 忘记密码需要输入账号、邮箱、验证码、新密码。
- 前端先校验新密码复杂度，再提交给后端。

#### Layout.vue

作用：

- 系统主布局。
- 左侧菜单和顶部用户信息。

实现方式：

- 根据当前用户角色显示不同菜单。
- 管理员显示用户管理、操作日志等菜单。
- 教师显示审核相关菜单。
- 学生显示我的预约、我的借用等菜单。

#### Dashboard.vue

作用：

- 首页统计。
- 展示实验室数量、设备可借数量、待审核数量、通过率、公告等信息。

实现方式：

- 页面加载时并发请求实验室、设备、预约、借用、公告数据。
- 前端根据返回数据计算统计卡片和简单图表。

#### Labs.vue

作用：

- 实验室信息页面。
- 管理员维护实验室，学生查看实验室。

实现方式：

### 2. lab 实验室表

保存实验室基础信息：

### 3. equipment 设备表

保存设备基础信息：

## 五、核心业务流程

### 1. 登录流程

```text
用户输入账号密码
-> 前端调用 login
-> 后端查询用户
-> 校验账号是否启用
-> 校验密码
-> 生成 token
-> 返回 token 和用户信息
-> 前端保存到 Pinia 和 localStorage
-> 跳转首页
```

### 2. 权限控制流程

```text
前端路由根据角色显示菜单
-> 请求接口时携带 token、用户ID、角色
-> 后端拦截器读取 token
-> 校验请求头身份和 token 身份一致
-> 根据角色判断接口是否允许访问
-> 允许则进入 Controller
-> 不允许则返回“当前角色无权执行此操作”
```

### 3. 实验室预约流程

```text
学生选择实验室、时间、用途
-> 前端提交预约申请
-> 后端校验申请人和实验室存在
-> 校验结束时间晚于开始时间
-> 校验该实验室该时间段没有冲突
-> 保存为 PENDING
-> 教师审核通过或驳回
-> 学生查看审核结果
```

### 4. 设备借用流程

```text
学生选择设备、数量、时间、用途
-> 后端校验设备存在
-> 校验借用数量大于 0 且不超过可借库存
-> 保存为 PENDING
-> 教师审核
-> 审核通过时扣减设备可借数量
-> 归还时恢复设备可借数量
```

### 5. 忘记密码流程

```text
用户输入账号和绑定邮箱
-> 后端校验账号邮箱匹配
-> 生成 6 位验证码
-> 存入验证码表
-> 发送邮件
-> 用户输入验证码和新密码
-> 校验验证码未过期、未使用
-> 校验新密码复杂度和不能等于旧密码
-> 更新密码
-> 标记验证码已使用
```

## 六、老师可能提问与回答

### 1. 为什么采用前后端分离？

可以回答：

> 前后端分离能让页面展示和业务接口解耦。前端负责交互和展示，后端负责业务规则和数据处理。这样结构更清晰，后期也方便扩展为移动端或其他客户端。

### 2. 你的权限是怎么控制的？

可以回答：

> 系统做了两层权限控制。前端通过路由和菜单控制不同角色能看到哪些页面；后端通过 RoleInterceptor 拦截接口请求，根据 token 中保存的用户角色判断接口是否允许访问。真正的安全控制在后端。

### 3. 预约冲突是怎么判断的？

可以回答：

> 我把预约看成时间区间。对于同一个实验室，如果已有预约的开始时间小于新预约的结束时间，并且已有预约的结束时间大于新预约的开始时间，就说明两个时间段有重叠，系统会拒绝提交。

### 4. 设备库存怎么保证正确？

可以回答：

> 学生提交申请时先检查借用数量不能超过可借数量。教师审核通过时扣减可借库存，归还时恢复库存。审核方法加了事务，保证借用记录状态和设备库存同时成功或同时回滚。

### 5. 为什么要有操作日志？

可以回答：

> 操作日志用于记录关键操作，比如用户管理、公告管理、预约审核、设备借用审核等。这样管理员可以追踪是谁在什么时间做了什么操作，提高系统可维护性和可审计性。

### 6. 密码是如何保存的？

可以回答：

> 密码不是直接明文保存，而是通过 PasswordUtil 使用 SHA-256 生成摘要后保存。系统还对新密码做了复杂度校验，要求至少 8 位并包含大小写字母、数字和特殊字符。

### 7. 邮箱验证码是怎么实现的？

可以回答：

> 用户输入账号和邮箱后，后端先校验账号邮箱是否匹配，然后生成 6 位随机验证码，保存到验证码表，并设置 5 分钟有效期。用户提交验证码和新密码时，后端校验验证码是否正确、未过期、未使用，然后更新密码。

### 8. 为什么学生只能看到自己的预约和借用？

可以回答：

> 查询预约和借用列表时，Controller 会根据请求头中的角色判断。如果是学生，就使用当前登录用户 ID 加查询条件，只返回自己的数据；教师和管理员可以查看全部记录。

### 9. 前端刷新页面后为什么还能保持登录？

可以回答：

> 登录成功后，前端把 token 和用户信息保存到 localStorage，同时 Pinia store 初始化时会从 localStorage 读取，所以刷新页面后仍然能恢复登录状态。

### 10. 你的项目还有哪些可以改进？

可以回答：

> 如果继续完善，可以把当前内存 token 改成 JWT 或 Redis；增加 Excel 导出；增加预约日历视图；增加消息提醒；部署到服务器并配置域名；同时可以增加更完整的单元测试和接口测试。

## 八、项目亮点总结

可以在答辩最后这样总结：

> 本系统围绕高校实验室预约和设备借用场景，实现了学生申请、教师审核、管理员维护的完整业务闭环。系统采用前后端分离架构，后端通过 Spring Boot 和 MyBatis Plus 实现接口和数据访问，前端通过 Vue 和 Element Plus 实现交互界面。项目中加入了角色权限控制、预约冲突检测、设备库存联动、邮箱验证码重置密码、操作日志、公告置顶下架等功能，能够满足课程项目对完整性和实用性的要求。
