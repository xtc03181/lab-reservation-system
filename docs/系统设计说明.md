# 高校实验室预约申请与设备管理系统设计说明

## 一、项目背景

高校实验室和实验设备是教学、实验和科研活动中的重要资源。传统管理方式通常依赖人工登记、线下沟通或表格统计，容易出现预约冲突、设备库存不清、审批过程不透明、历史记录难追踪等问题。

本系统围绕“实验室预约申请”和“设备借用管理”两个核心场景，设计并实现一个基于 Web 的管理系统，使学生、教师审核员和管理员可以在线完成预约、借用、审核、公告查看和系统维护等操作。

## 二、设计目标

系统设计目标如下：

- 实现实验室预约和设备借用的线上化管理。
- 实现学生申请、教师审核、管理员维护的业务闭环。
- 支持不同角色登录后看到不同菜单和功能。
- 支持管理员配置预约规则，包括提前预约天数、单次预约时长、每日预约次数和周末预约限制。
- 支持管理员配置实验室开放星期和开放时间，学生只能在开放时间内预约。
- 支持预约冲突检测，避免同一实验室同一时间段重复预约。
- 支持设备库存联动，审核通过后扣减库存，归还后恢复库存。
- 支持公告管理、操作日志、数据导出、日历视图和审批提醒。
- 支持站内消息中心，向教师推送待审核消息，向学生推送审核结果。
- 使用 Redis 对公告列表进行缓存，提高高频查询效率。
- 使用 GitHub 进行版本备份，方便项目管理。

## 三、技术架构

本项目采用前后端分离架构。

### 1. 前端技术

- Vue 3：构建前端单页面应用。
- Element Plus：提供表单、表格、弹窗、按钮、标签、分页等 UI 组件。
- Axios：负责 HTTP 请求。
- Pinia：保存登录用户和 token 状态。
- Vue Router：管理页面路由和前端权限。
- Vite：前端开发和构建工具。

### 2. 后端技术

- Spring Boot：后端应用基础框架。
- Spring MVC：提供 REST 接口。
- MyBatis Plus：简化数据库 CRUD 操作。
- MySQL：保存业务数据。
- Redis：缓存公告列表。
- JavaMailSender：发送邮箱验证码。
- Maven：后端项目构建工具。
- 内置 Tomcat：运行 Web 服务。
- Docker Compose：统一编排 MySQL、Redis、后端和前端容器。

### 3. 数据传输方式

前后端通过 JSON 进行数据传输。

前端请求：

```text
Vue 页面 -> api/modules.js -> Axios -> 后端 REST 接口
```

后端响应：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

## 四、系统总体流程

```text
用户登录
-> 后端校验账号密码
-> 返回 token 和用户信息
-> 前端根据角色显示菜单
-> 用户进行预约、借用、审核、公告等操作
-> 后端拦截器校验 token 和角色权限
-> Service 层处理业务逻辑
-> Mapper 层访问 MySQL
-> Redis 缓存高频公告数据
-> 返回处理结果
```

## 五、角色设计

### 1. 学生

主要功能：

- 查看实验室信息。
- 查看设备信息。
- 提交实验室预约申请。
- 修改或取消自己的待审核预约。
- 提交设备借用申请。
- 查看自己的预约和借用审核结果。
- 查看公告。
- 修改个人资料和密码。

### 2. 教师审核员

主要功能：

- 查看待审核预约申请。
- 审核实验室预约。
- 查看待审核设备借用申请。
- 审核、驳回或确认归还设备借用。
- 查看公告。
- 查看首页统计和待办提醒。

### 3. 管理员

主要功能：

- 维护实验室信息。
- 配置实验室开放星期和每日开放时间。
- 维护设备信息。
- 管理用户。
- 管理公告，包括发布、编辑、置顶、下架、删除。
- 查看操作日志。
- 查看全部预约和借用记录。
- 查看系统统计数据。

## 六、后端分层设计

后端主要分为以下层次：

### 1. Controller 层

作用：

- 接收前端 HTTP 请求。
- 获取请求参数和请求头。
- 调用 Service 层处理业务。
- 返回统一的 `ApiResponse`。

主要 Controller：

- `AuthController`：登录和密码重置。
- `UserController`：用户管理和个人中心。
- `LabController`：实验室管理。
- `EquipmentController`：设备管理。
- `LabReservationController`：实验室预约。
- `EquipmentBorrowController`：设备借用。
- `NoticeController`：公告管理。
- `OperationLogController`：操作日志。

### 2. Service 层

作用：

- 实现核心业务规则。
- 校验业务数据是否合法。
- 调用 Mapper 访问数据库。
- 处理事务和缓存。

典型业务：

- 登录校验和 token 生成。
- 预约时间冲突判断。
- 设备库存扣减和恢复。
- 密码复杂度校验。
- Redis 公告缓存。

### 3. Mapper 层

作用：

- 负责数据库访问。
- 通过 MyBatis Plus 提供基础 CRUD 能力。

### 4. Entity 层

作用：

- 映射数据库表结构。
- 每个实体类对应一张数据表。

### 5. Common 公共模块

作用：

- 统一响应格式。
- 统一异常处理。
- 密码工具类。
- token 存储。

## 九、核心业务设计

### 1. 登录与权限

用户登录成功后，后端生成 token，前端保存 token 和用户信息。

后续请求中，Axios 请求拦截器自动携带：

- `Authorization`
- `X-User-Id`
- `X-User-Role`

后端 `RoleInterceptor` 校验 token 和角色，防止未登录访问和越权访问。

### 2. 预约冲突检测

系统判断同一实验室的时间段是否重叠。

判断条件：

```text
已有预约开始时间 < 新预约结束时间
并且
已有预约结束时间 > 新预约开始时间
```

如果条件成立，说明两个预约时间段冲突。

### 3. 预约规则配置

管理员可以在后台配置预约规则：

- 最多提前几天预约。
- 单次预约最长几个小时。
- 每人每天最多预约几次。
- 是否允许周末预约。

学生提交或修改预约时，后端会读取当前启用规则并进行校验。如果不符合规则，系统会返回明确提示。

### 4. 设备库存联动

设备借用申请提交时，系统校验借用数量不能超过可借数量。

教师审核通过后：

```text
设备可借数量 = 当前可借数量 - 借用数量
```

设备归还后：

```text
设备可借数量 = 当前可借数量 + 借用数量
```

审核过程使用事务，保证借用记录和设备库存一致。

### 5. Redis 公告缓存

公告属于读多写少的数据，因此适合缓存。

查询公告时：

```text
先查 Redis
-> 命中则直接返回
-> 未命中则查 MySQL
-> 将查询结果写入 Redis
```

缓存 key：

- 管理员公告：`lab-reservation:notices:admin`
- 普通用户公告：`lab-reservation:notices:public`

缓存有效期：

```text
10 分钟
```

当公告发布、编辑、置顶、上架、下架或删除时，系统清除 Redis 缓存，保证下次查询的是最新数据。

### 6. 操作日志

系统在关键操作后记录日志：

- 用户管理
- 个人资料修改
- 密码修改
- 预约提交、修改、取消、审核
- 设备借用提交、审核、归还
- 公告发布、编辑、置顶、下架、删除

日志用于追踪用户行为，方便管理员审计。

### 7. 消息中心

系统提供站内消息中心，用于主动提醒用户处理事项。

消息触发规则：

- 学生提交实验室预约后，教师审核员和管理员收到待审核消息。
- 教师审核实验室预约后，学生收到审核结果消息。
- 学生提交设备借用申请后，教师审核员和管理员收到待审核消息。
- 教师处理设备借用后，学生收到处理结果消息。

每条消息包含：

- 接收人
- 标题
- 内容
- 消息类型
- 是否已读
- 创建时间

前端在顶部和侧边栏展示未读消息数量，用户可以进入消息中心查看并标记已读。

## 十、安全设计

系统安全措施包括：

- 登录 token 校验。
- 后端角色权限拦截。
- 密码 BCrypt 加密存储。
- 新密码复杂度校验。
- 邮箱验证码重置密码。
- 验证码有效期和使用状态控制。
- 用户密码字段返回前置空，避免传给前端。
- 操作日志记录关键行为。
- 兼容旧明文密码和旧 SHA-256 密码，登录成功后自动升级为 BCrypt。

后续可优化：

- 使用 JWT 或 Redis Session。
- 增加登录失败次数限制。
- 部署 HTTPS。
